# Contributing

Спасибо за интерес к `gost-crypto`. Ниже — правила, которые упростят
ревью и ускорят мердж.

## С чего начать

1. Форкните репозиторий и создайте ветку: `feature/xxx` или `fix/yyy`.
2. Убедитесь, что всё собирается и тесты проходят:
   ```bash
   go build ./...
   go vet ./...
   go test -count=1 ./...
   ```
3. Для криптографических изменений — прогоните race-детектор:
   ```bash
   go test -race -count=1 ./...
   ```

## Стиль кода

- `gofmt` обязателен. PR с неотформатированным кодом возвращаются без ревью.
- Публичный API минимален: экспортируйте только то, без чего пользователю не обойтись.
- Именование пакетов — по номеру стандарта (`gost341112` = ГОСТ Р 34.11-2012).
- Каждый `.go` файл начинается с AGPL-3.0 copyright header (см. существующие файлы).
- Комментарии для `go doc` — на английском; комментарии внутри реализации — по месту.

## Тесты

- Для каждого алгоритма **обязательны** KAT-векторы из официального стандарта или RFC.
- KAT оформляются как table-driven тесты, со ссылкой на источник в комментарии.
- Для каждого алгоритма **обязателен** бенчмарк (`Benchmark*`).
- Регрессии на производительность и аллокации обсуждаются отдельно в описании PR.

## Безопасность

- Никаких внешних зависимостей. Всё — поверх stdlib.
- Ключевой материал должен обнуляться после использования (см. `internal/subtle`).
- Любой путь, зависящий от секрета, должен быть constant-time.
- Уязвимости сообщайте приватно — см. [SECURITY.md](SECURITY.md).

## Коммиты и PR

- Атомарные коммиты. Одно логическое изменение — один коммит.
- Сообщение коммита: короткий заголовок + пояснение «зачем», если не очевидно.
- PR в `main`. В описании PR — что, зачем, какие тесты/KAT добавлены.
- CI должен быть зелёным перед запросом ревью.
